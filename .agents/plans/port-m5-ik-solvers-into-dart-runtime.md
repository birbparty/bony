## Change Information

### Change Type
MIGRATION — port of the M5 IK solver + evaluation behavior from the Nim reference runtime into the Dart runtime. No new capability is invented; the Nim implementation is the sole behavioral authority and the committed numeric goldens are the cross-runtime contract. This is Dart-runtime + Dart-test + doc-status only.

### Description
Port the M5 IK solvers into the Dart runtime so `computeWorldTransforms` evaluates IK constraints at pose time (matching the Nim reference), and add the m5_ik setup-pose golden to the Dart cross-runtime conformance gate.

The M5-IK format, the Nim reference solver, and the cross-runtime numeric goldens have all landed. The Dart runtime currently only *loads* IK constraint data (`IkConstraintData` + `SkeletonData.ikConstraints`) and returns the unconstrained setup-pose hierarchy; `docs/ik-constraint-format-contract.md` §8 marks Dart IK evaluation as "deferred to a later slice." This milestone closes that gap: implement the three IK solvers in Dart, wire them into the world-transform pass mirroring the existing path-constraint evaluation machinery, and match the committed golden `conformance/goldens/m5_ik_rig_t0.json` within the `1e-4` cross-runtime tolerance.

**The Nim reference to mirror (port its behavior — do not re-derive geometry, and do not invent a different solver, mix convention, quantization point, or anchoring model):**

- **Solvers** — `runtime-nim/src/bony/constraints/ik.nim`. No Dart equivalent exists yet (confirmed: no `solve*` functions or `*IkResult` types under `runtime-dart/`). Port into a NEW `runtime-dart/lib/src/ik.dart`:
  - `solveOneBoneIk` (ik.nim:84), `solveTwoBoneIk` (105), `solveChainIk` (155), and result types `OneBoneIkResult` / `TwoBoneIkResult` / `ChainIkResult` (ik.nim:17-30).
  - FABRIK numerics: `fabrikIterations = 8`, `fabrikTolerance = 1e-4` (ik.nim:8-9), `solverEpsilon = 1e-12` (ik.nim:10), the collinear/degenerate bend-plane seeding (ik.nim:183-200) and the root-past-total-length straight-line case (ik.nim:175-181). Port these exactly — they are load-bearing for the chain case.
  - The two-bone convention: `solveTwoBoneIk`'s `childRotation` input is RELATIVE to the parent, and its result `childRotation` is the child's angle relative to the parent (so the child's absolute world angle is `parentRotation + childRotation`). `solveOneBoneIk` and `solveChainIk` return/emit ABSOLUTE world angles.
  - **Quantization — mirror Nim exactly; do NOT over-quantize (verified against source by two independent reviews).** `ikPoint(x, y)` (ik.nim:33-34) quantizes both coords via `quantizeF32`, but the runtime evaluation path (`applyRuntimeIk`) does **not** use it: it constructs `target`, `restOrigins`, `currentOrigins`, and the chain `points` with the direct, non-quantizing `IkPoint(x: …, y: …)` constructor (transform.nim:419, 430, 457, 497). The solvers' `requirePoint` (ik.nim:55-59) only checks finiteness — it does not quantize. The committed golden was generated by that unquantized runtime path, so **Dart must construct those points WITHOUT quantization** — adding `quantizeF32` at those sites would make Dart *diverge* from the golden, not match it. Port `ikPoint` WITH quantization ONLY as the solver-input constructor used by hand-authored solver unit tests (the `test_smoke.nim` parallel); inside `_applyRuntimeIk`, construct points directly from world `tx`/`ty` exactly as Nim does. Dart already has `quantizeF32` in `runtime-dart/lib/src/deform.dart:13` (it takes no `context` argument — that Nim parameter is an error-message detail with no Dart equivalent and can be dropped). **NOTE for implementers:** the originating `/big-change` prompt instructed quantizing `target`/`restOrigins`/`currentOrigins`; that instruction was checked against the Nim source and found incorrect. The conformance note's "only IK point inputs are f32-quantized internally" refers to the `ikPoint` test-input helper, not the runtime path. If in doubt, mirror `applyRuntimeIk` line-for-line and confirm empirically against `m5_ik_rig_t0.json`.

- **Evaluation entry point + geometry contract** — `applyRuntimeIk` (`runtime-nim/src/bony/transform.nim:389-524`). Load-bearing rules:
  - **Fixed segment lengths from the REST pose** (`restWorldFor`, transform.nim:357; `ikDistance`, 353), but the chain **anchors at the CURRENT (live) joint origins** so a moved parent is tracked — this is the me5.13 "current-pivot anchoring" contract decision (§4). `mix` is applied ONCE inside the solver, so `mix = 0` is the current-pose identity.
  - The solver reads the bones' CURRENT world rotations (`worldRotationDegrees`, transform.nim:349) and the target's CURRENT world position. For the 2-bone case the child rotation input is RELATIVE to the parent (current child world rotation minus current parent world rotation); 1-bone and chain solvers return ABSOLUTE world angles. The write-back normalizes to absolute (`solvedWorldAngles`, transform.nim:462-503).
  - The chain root's CURRENT world is FK-composed from bone[0]'s external parent forward (transform.nim:436-449); when a chain bone's parent is the previous chain bone, reuse the already-computed chain world rather than reading the global `worlds` array.
  - **Ordering:** the target (and any external chain-root parent) must already be computed before the constraint runs; otherwise raise the Dart equivalent of Nim's `orderingViolation` (transform.nim:418, 447) — reuse the same ordering model as `_applyRuntimePathConstraint` (which throws a `FormatException` on a not-yet-computed target/parent).
  - **Sequential FK write-back** (transform.nim:505-524): for each chain bone, convert its solved absolute world angle to the bone's LOCAL rotation against its already-re-worlded parent (subtract parent world rotation ONLY when the bone `inheritRotation` is true), set the local rotation, re-world the bone, mark computed. A bone that does not inherit rotation has world rotation equal to its own local rotation (subtract nothing).

**The Dart seams to extend (all in `runtime-dart/lib/src/transform.dart`):**

- `computeWorldTransforms` (transform.dart:448) currently branches only on `hasRuntimePaths` (line 450). Widen the gate to path-OR-ik: `data.paths.any((p) => p.runtimeEvaluable) || data.ikConstraints.any((c) => c.runtimeEvaluable)`, mirroring Nim's `hasRuntimeConstraints` loop (`transform.nim:137-147`).
- **Build ONE unified constraint update cache**, not two passes. Dart's current `_buildPathConstraintUpdateCache` (transform.dart:280-368) is path-only — it collects only `data.paths` and sorts by `(order, sourceIndex)`. Generalize it to collect BOTH `path.runtimeEvaluable` and `ik.runtimeEvaluable` constraints into a single ordered list, and generalize `_ConstraintEntry` (transform.dart:59-62, currently carries only `sourceIndex`) to carry a constraint **kind** tag (path vs ik) alongside `sourceIndex`. The ordering is binding and defined by Nim's `compareConstraintEntries` + `constraintKindRank` (`runtime-nim/src/bony/model.nim:715-734`) and `docs/constraint-total-order.md`: for the path+ik subset both share stage rank 0, so sort by `order` value first, then by kind rank where **ckIk (0) precedes ckPath (2)** on a tie, then by source index. Do NOT iterate paths and IK in separate loops — a dual-loop gets the tie order wrong (an IK and a path both at `order = 0` must run IK first) and breaks the golden.
- The write-blocker / bone-group interleave logic (transform.dart:305-366) must account for IK: an IK constraint WRITES all of `ik.bones` (not a single `path.bone`). Set write-blockers for **every** IK chain bone. For the READ set, mirror Nim's cache exactly: Nim lists only `@[ik.target]` as the constraint's reads, so the cache should walk the `ik.target` ancestor lineage and emit it before the constraint (the same lineage walk the path path does for `path.target` at transform.dart:330-347). The chain root's **external parent** is NOT enumerated as a cache read — Nim enforces it with a *runtime* ordering check inside `applyRuntimeIk` (raising `orderingViolation` at transform.nim:446-447), so replicate that as a `FormatException` in `_applyRuntimeIk`, not as extra cache-lineage logic. Do not over-build the cache to walk external-parent lineage.
- Then dispatch each `_ConstraintEntry` by kind to `_applyRuntimePathConstraint` or the new `_applyRuntimeIk`, exactly as Nim's cache dispatches `ckPath`/`ckIk` (`transform.nim:167-176`).
- Reuse `_BoneGroupEntry` (transform.dart:54) for the bone-group entries interleaved between constraints — already kind-agnostic. Reuse `_worldForBone` (114) for re-worlding, `_lerp` (191), and the `_Point`/`_distance` helpers (30-33, 187). Add a Dart `worldRotationDegrees` equivalent (Nim transform.nim:349) — none exists yet in the Dart runtime.
- `IkConstraintData` (`runtime-dart/lib/src/model.dart:90-119`) already exposes `bones`, `target`, `mix`, `bendPositive`, `order`, and the `runtimeEvaluable` getter (line 118). Its doc comment currently says "Dart evaluation is deferred … this getter exists for model parity only" — update that comment to reflect that Dart now evaluates IK. Read that class for the exact field set before coding (note `mix` and `bendPositive` are nullable; absent `mix` defaults to 1.0, absent `bendPositive` defaults to true). Null-coalesce explicitly at the call site — pass `mix ?? 1.0` into the solver and derive `bendSign = (bendPositive ?? true) ? 1.0 : -1.0` — never hand a raw nullable to the solver.

## Bead Creation Guidelines

### Epic / Hierarchy (REQUIRED)
- Create exactly **one parent epic** for the whole change: `EPIC=$(bd create "Epic: <change summary>" -t epic -p 0 --label epic --silent)`.
- Parent **every** task bead to it: add `--parent "$EPIC"` to every `bd create` (children inherit the epic's labels unless you pass `--no-inherit-labels`).
- The epic is a **rollup, not work**: never `bd dep add` to or from it. Membership is `--parent`; `bd dep add` is reserved for real ordering edges *between task beads*. A blocking edge on an epic wrongly keeps it out of (or drops it into) `bd ready` and inverts `bd dep tree`.
- **Keep the epic out of `bd ready`** by marking it active right after creation: `bd update "$EPIC" --status in_progress`. `bd ready` excludes `in_progress`/`blocked`/`deferred`/`hooked`. Do **not** rely on `--exclude-type epic` — that flag is ineffective on some `bd`/`bn` builds, whereas status-based exclusion works everywhere.
- An epic must have **≥ 2 children** to be meaningful — a one-task change does not need this skill.
- For very large changes you MAY use phase sub-epics (each `--parent "$EPIC"`, each with its own children), but a single top-level epic is the default and is sufficient for most changes.

### Priority Levels
- `-p 0` = Critical (blocking other work, or high-risk changes needing early validation)
- `-p 1` = High (important implementation work)
- `-p 2` = Medium (standard work)
- `-p 3` = Low (cleanup, nice-to-haves)

### Labels (Phase Grouping)
Use `--label` to group beads by phase:
- `analysis` - Understanding current state
- `prep` - Preparation work (characterization tests, feature flags, scaffolding)
- `impl` - Core implementation
- `testing` - Test coverage
- `migration` - Data/code migration
- `docs` - Documentation updates
- `cleanup` - Post-rollout cleanup

### Dependency Rules
1. Never create cycles
2. Analysis tasks should complete before implementation begins
3. Characterization tests should exist before changing code
4. Use `bd dep add CHILD PARENT` (child depends on parent completing first)
5. Parallel work should share a common ancestor, not depend on each other
6. `bd dep add` is for ordering edges **between task beads only** — never use it to attach a task to the epic (that is `--parent`), and never add a blocking edge to or from the epic

### Task Granularity
- Each bead should be completable in **under 750 lines of code changed**
- Tasks should be atomic enough for one agent to complete without coordination
- If a task requires multiple file areas, consider splitting by file area
