**Description**

Prompt 09 made transform constraints a first-class format record evaluated by the
Nim reference runtime and produced the numeric golden
`conformance/goldens/m5_transform_rig_t0.json`. The Dart runtime currently
evaluates IK and path constraints but has **zero** transform-constraint support
(`grep -rilE 'transformConstraint' runtime-dart/lib/` returns nothing). This
prompt ports transform-constraint evaluation to Dart and gates it in the Dart
conformance suite, exactly mirroring the completed Dart IK evaluation slice
(bead bony-7b7 / prompt `08-runtime-dart-ik-evaluation.md`).

Work items:
1. **Loader**: parse the new `transformConstraints` array into the Dart model.
   Mirror how `ikConstraints` are parsed in `runtime-dart/lib/src/loader.dart`
   and represented in `runtime-dart/lib/src/model.dart`. The generated
   `runtime-dart/lib/src/generated/wire.dart` (regenerated by prompt 09) already
   carries the wire keys — do not hand-edit it.
2. **Solver + evaluation**: port the transform-constraint solver
   (`affineToTransformPose` / `transformPoseToAffine` / `applyTransformConstraint`
   with per-channel translate/rotate/scale/shear mix) from
   `runtime-nim/src/bony/constraints/transform_constraints.nim` into Dart —
   project-owned affine math, ported symbol-for-symbol from the Nim reference, not
   from any third party. Wire it into the constraint dispatch inside
   `computeWorldTransforms` (`runtime-dart/lib/src/transform.dart`) so transform
   constraints evaluate at their correct position in the unified constraint order.
   The Nim total order is `ckIk (0) → ckTransform (1) → ckPath (2) → ckPhysics (3)`
   (`runtime-nim/src/bony/model.nim:715-720`). Dart today defines only
   `enum _ConstraintKind { ik, path }` with `_constraintKindRank` mapping
   `ik => 0, path => 2` (rank slot `1` is deliberately left open, with a comment
   that transform/physics are out of scope) — `runtime-dart/lib/src/transform.dart:60-68`.
   So the work is: add a `transform` enum variant at rank `1`, add a
   transform-constraint descriptor-collection loop mirroring the existing IK/path
   loops (with `writes` = constrained bone, `reads` = target), add a
   `_applyRuntimeTransformConstraint`, and add the dispatch switch case — the
   shared sort/cache builder itself needs no restructuring. `ik_constraint_order_test.dart`
   exercises the existing IK-before-path tie-break and is the pattern to extend.
3. **applyPose preservation**: ensure transform constraints survive any
   `SkeletonData` rebuild in `runtime-dart/lib/src/anim.dart`. The Dart IK story
   slice (bead bony-1c5) fixed a bug where `applyPose` dropped `ikConstraints` when
   rebuilding `SkeletonData`, silently losing all IK on any posed/animated skeleton.
   Do not reintroduce that class of bug for transform constraints — if `applyPose`
   reconstructs `SkeletonData`, it must carry `transformConstraints` through.
4. **Conformance gate**: add an `m5_transform_rig` numeric-golden group to
   `runtime-dart/test/m10_conformance_test.dart` (mirror the existing M5-IK group)
   asserting the Dart `computeWorldTransforms` output matches
   `conformance/goldens/m5_transform_rig_t0.json` within `1e-4` for every bone
   world affine (and draw batches if the golden carries them). Add a focused
   transform-constraint solver/order unit test mirroring
   `runtime-dart/test/ik_constraint_order_test.dart` /
   `runtime-dart/test/ik_solver_test.dart` if the existing tests do not already
   cover the new path.
5. **Docs**: update the cross-runtime status note in `conformance/README.md` for
   the transform rig (state that the setup-pose golden is now honored by both the
   Nim reference and the Dart runtime, mirroring the IK status line). If prompt 09
   left any conformance README status text saying transform constraints are
   Nim-only, flip it to reflect Dart parity.
